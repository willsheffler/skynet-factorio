## Top finding: Factorio Learning Environment (FLE)

- **Repo:** https://github.com/JackHopkins/factorio-learning-environment
- **License:** MIT
- **Paper:** [arXiv:2503.09617](https://arxiv.org/abs/2503.09617) (NeurIPS 2025 Datasets and Benchmarks track)
- **Maintenance:** v0.4.3 released 2026-04-06; 985 GitHub stars; 879 commits on main; actively maintained.
- **Stack:** Python 3.10+, Docker, Factorio 2.0.73+ (optional, for rendering)
- **Install:** `pip install factorio-learning-environment` → `fle cluster start` → `fle eval`

### What it provides

A REPL-style observation/action loop for LLM agents in Factorio:

1. Agent observes structured game state
2. Generates Python program as action
3. Action executes against Factorio via FLE's runtime
4. Receives execution result + new state
5. Loops

This is exactly the brain/observation/action layer skynet-factorio needs. The
LLM-agent contract is well-tested by FLE's NeurIPS-published benchmark.

### Gap from our use case

FLE is single-player evaluation focused. It runs Factorio in Docker for
controlled benchmark conditions. Multiplayer co-play (human + bot in shared
game) is NOT in scope.

### Borrow strategy

- Adopt FLE's observation schema + action API verbatim where it fits.
- Replace FLE's Docker-orchestrated single-player runtime with our
  headless-hosts-multiplayer topology.
- Keep FLE's brain pluggable so we can swap LLM choice later (Claude, local
  Qwen, OpenAI).

## Other relevant projects

### Windfisch/factorio-bot
- **Repo:** https://github.com/Windfisch/factorio-bot
- **License:** GPL-3.0 (+ MIT for portions)
- **Stack:** C++17 (84.8%), Lua (10.9%), shell scripts
- **Maintenance:** Early-stage POC.
- **Architecture:** External C++ bot connects via RCON; Lua mod handles in-game integration.
- **Multiplayer:** Limited testing ("second player following remote-controlled one will not see anything jumpy"). Not formal co-play.

GPL license means we can read for reference but should avoid copy-pasting into our MIT-or-similar codebase. Useful reference for multiplayer mechanics, esp. the Lua mod side and the "fair info" principle (bot only uses info a human would have).

## Architecture decision: headless hosts the game

Original Will hypothesis: bot joins Will's hosted game. Cleaner approach
after looking at how multiplayer mods work: **bot's headless Factorio HOSTS
the game; Will and Jonah connect as clients.**

Reasons:
- Bot's Lua mod runs in-process with the simulation that holds canonical state.
- No LAN-protocol layer between bot's decision-making and game state.
- Standard multiplayer-mod assumption (bot mod loads with the hosting instance).
- Avoids the "AI teammate in multiplayer" desync issues that bit AI Player mod.
- Will + Jonah's clients are vanilla Steam Factorio with no mod requirement.

Cake runs both Will's Steam GUI Factorio AND the headless server side-by-side
(confirmed feasible — they're separate processes with separate data dirs and
ports). Bot RCON port is loopback-only.
